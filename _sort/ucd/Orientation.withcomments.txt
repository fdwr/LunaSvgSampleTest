# ================================================
# Generation rules:
#
# If Latin, Greek, Cyrillic (and not symbol other, like U+0482 CYRILLIC THOUSANDS SIGN)
# then stackable, mirrorable, rotatable, and discrete.
#
# If symbol other and script common, 
# then stackable, rotatable, and discrete.
#
# If full width (even if Latin), mark as upright (not rotatable) (but still boustrophodonic mirrorable)
# You can almost make a firm statement that if the codepoint is any of the following scripts, then it remains upright: Bopomofo, Han, Hangul, Hiragana, Katakana, Yi
# However, there are half-width Katakana and Hangul characters that are not upright (they have a ‘narrow’ decomposition type which can be the key to this exception).
# For many Common script, you should be able to use the ideographic line breaking class (ID) to determine uprightness. Though, abusing the line breaking class this way seems somewhat unreliable, and not future-proof (but maybe fine in practice).
# For remaining Common script, you could look for ambiguous ideographic (AI), but there are many symbols that are instead alphabetic (AL) which should be upright. For example, you want to keep the Victory Hand and Latin Cross upright, but they are both Common AL. You also may not want ‘Three Rightward Arrows’ to suddenly become ‘Three Downward Arrows’ (another Common AL). Roman Numerals actually seem like great candidates for stacking (AI), but they are script Latin, so which wins?
# If any new symbols have mirror pairs, mark as rotatable, mirror paired (or DSR if ideographic).
#
# Equivalence Classes:
#
# GDI and GDI+ only have two classes:
#
# 1. Characters that remain upright.
# 2. Characters that follow the baseline.
#
# For supporting stacked text or the other proposed UAX #9 approach,
# two classes is insufficient. Though, for the simpler client who only
# wants the GDI-like approach, all the classes below can be mapped to
# these two classes.
#
# In addition to rotation, the class data contains enough information
# to distinguish mirroring characters and scripts too. Most mirrorable
# characters (notably common punctuation) already have a substitution
# available via character translation; but a handful of scripts also
# mirror based on direction, and these are unlikely to ever receive
# (nor should they) mirror codepoints. Instead, these should be
# mirrored via a graphic transform or font feature, such as rtlm. The
# data does not specify which should be used.
#
# D  - discrete (generally freely rotatable and stackable)
# C  - cursive/connected (not stackable, as it would look broken)
# S  - stackable
# R  - rotatable
# RV - rotatable via vert feature
# M  - mirrorable
# MP - mirrorable via mirror pair
# B  - baseline following
# 
# - Mirrorable characters that have a mirror pair ({ } < >).
# - Mirrorable characters that have no mirror pair (but achievable for L bidi class via rtlm, or R bidi class via ltrm).
# - Should always remain upright, regardless of reading direction. These never mirror, regardless of reading direction (Chinese/Korean/Japanese). The only exception may be coercing them bottom-to-top, where it’s more sensible to rotate them 270CW.
# - Should remain oriented along the baseline following the reading direction. These rotate in vertical and via a transform (90CW for ttb along with Latin, rotated 270CW for btt) (most opening/closing parenthesis)
# - Should remain oriented along the baseline (rotated 90CW for ttb, rotated 270CW for btt). However, these are historically rotated via font’s ‘vert’ feature rather than a graphic transform. Using a graphic transform to accomplish rotation instead would actually produce quite reasonable for most of these characters (if the font lacked a vert feature), but ideally, the font substitutes different forms with an adjusted width and height (lest the CJK font have a non-square character ratio) and stem width. Note that punctuation such as the ideographic full-stop (U+3002 “。”) are not part of this class. They require the vert feature always to be placed correctly within the square (and are simply grouped with the other kana).
# 
# Should (or rather, may) orient to follow the reading direction (Mongolian in RTL, Ogham in RTL?)
# Mirrorable scripts like Runic/Egyptian hieroglyphics/Mayan that mirror according to reading direction and always stack in top to bottom (not rotate).

# DiscreteS: Han, Hangul, Japanese Katakana & Hiragana, Yi, Phags-Pa(?)
# Stackable, doesn't rotate in TTB, doesn't mirror, goes LTR generally, but occasionally RTL (for single column height wrapped text on signs, or sometimes on the sides of vehicles).
# This also includes some other scripts that generally rotate, such as Latin full-width characters: U+FF21 FULLWIDTH LATIN CAPITAL LETTER A.

# DiscreteSRM_LTR: Latin, Greek, Cyrillic, also including Gothic, Shavian (maybe U+222B INTEGRAL?)
# Stackable, rotatable.
# Although not found today as RTL, these inherit an ancient legacy of mirrored boustrophedon, such that if they were coerced into RTL, each glyph would mirror. The only time you ever see this nowadays is on ambulances, and that's only so that you see it correctly in your mirror.

# DiscreteSR: Cherokee(L), Braille(L), Thai(L), Khmer(L), Cypriot(R), Hebrew(R), Saurashtra(L), Gujarati(L), NKo(R), Sundanese(L)
# Stackable and rotatable, but does not mirror. Most of the remaining discrete scripts fall in here.

# DiscreteSRM_RTL: NA

# DiscreteSR_LOR: most punctuation
# This punctuation can either stack or rotate, such as the exclamation mark or commercial at sign. It is direction neutral and does not mirror.

# ConnectedR_LOR: ...
# Most scripts and any unknown scripts. They are assumed to not mirror and rotate (to keep any necessary glyph connection).

# ConnectedR_LTR: Devanagari, Gurmukhi, Bengali, Malayalam, Telugu, Syloti_Nagri
# Always rotates, never mirrors or stacks. Rotates CW90 for TTB. Keeps orientation in LTR (rather than rotating upside down). Note that Malayalam and Telugu are not actually 'connected' scripts in that the glyphs actual touch each other, but the writing is left connected along the baseline (otherwise the variable width clusters would look odd stacked). It may be possible to stack them reasonably by stacking individual glyphs rather than whole glyph clusters, but that's brittle.

# ConnectedR_RTL: Arabic, Syriac
# Always rotates, never mirrors or stacks. Rotates CW270 for TTB (or CW90 for TTB with bidi level odd), CW90 for BTT. Keeps orientation in LTR (rather than rotating upside down).

# ConnectedTTB: Mongolian
# Always rotates, never mirrors or stacks. Strongly TTB in natural orientation, but rotates CCW in LTR (or validly CW in RTL).

# ConnectedBTT: Ogham
# Always rotates, never mirrors or stacks. BTT in natural orientation, but curves around to TTB too. LTR is 0CW. RTL is 180CW.

# DiscreteSM_LOR_TTB: Egyptian Hieroglyphics(L), Runic(L), Mayan glyphs, Sabean(R)
# Stackable, never rotates in TTB, goes left or right, mirrors in RTL-TTB (otherwise would be retrograde reading). Also mirrors horizontally for TTB-RTL (identity for TTB-LTR).

# DiscreteR_LTR: Rongo Rongo (rotating boustrophedon)
# Default LTR. Rotates 180CW in RTL. Rotates 90CW in TTB, 270CW in BTT (though seems stackable would be valid too).
# To read a rongo-rongo tablet, you start at the bottom left-hand corner and read along the line. Then you turn the tablet by 180 degrees and begin reading the next line up, again from left to right. At the end of that line, you repeat the 180-degree turn, and so on.)

# SymbolRMP_LTR: Opening/closing symbols
# These should rotate in both TTB rotating and TTB stacked layout, such as the parenthesis U+0028. They also mirror, but via a character mirror pair (not via a font feature or transform, though, those would work too).

# SymbolRF: Ideographic opening/closing symbols
# These should also rotate, but they are generally rotated via the font's 'vert' feature. Thus the orientation from layout's point of view is actually 0CW (otherwise double rotation would occur). If the font lacked a vert feature for that character, the character could be reasonably rotated via a graphic transform (the stem width and placement will less ideal than if the vert feature existed, but it will be an improvement over remaining upright). Examples include U+3014 LEFT TORTOISE SHELL BRACKET and U+30FC KATAKANA-HIRAGANA PROLONGED SOUND MARK.

# Arrow_SMR: Arrows
# These can technically always remain fixed to the original direction, mirror, or rotate to follow the baseline. The default is to remain upright (0CW), regardless of reading direction.

#         RF (full-width)
